# 코딩 스타일 표준 (Coding Styles)

**ID:** `SKL-CODING-STYLES`  
**범위(Scope):** AI Recommended  
**우선순위:** High  
**적용 조건:** 명명 케이스·포맷·주석·커밋 메시지 규약을 다루는 범용(foundational) 코딩 스타일 표준으로, 특정 언어/프레임워크에 무관하다. 새 코드를 작성하거나 스타일·명명·포맷·커밋 메시지 규칙을 정·정비할 때 읽는다. 브랜치/PR/머지 등 Git 협업 흐름 자체는 git-workflow 스킬을 본다.

---

## 지시사항 (Instructions)

1. 일관성이 곧 가독성: 개인 취향보다 팀이 합의한 한 가지 방식을 따른다. '맞는 스타일'보다 '같은 스타일'이 중요하다.
2. 명명은 의도를 드러낸다: 이름만 보고 무엇인지·어떤 종류인지 알 수 있게 한다. 대상 종류(변수·함수·타입·상수·파일)별로 정해진 케이스를 일관되게 적용한다.
3. 포맷은 도구로 강제한다: 들여쓰기·세미콜론·따옴표 같은 포맷은 사람이 눈으로 맞추지 말고 포매터/린터로 자동 정렬한다. 리뷰에서 포맷을 논쟁하지 않는다.
4. 주석은 '왜'를 적는다: 코드가 말해주는 '무엇'을 반복하지 말고, 코드로 드러나지 않는 의도·배경·트레이드오프를 적는다.
5. 커밋 메시지는 규약을 따른다: 메시지 첫 줄을 정해진 타입 규약(예: feat/fix/...)으로 시작해, 이력만 훑어도 변경 성격이 보이게 한다.
6. 스택 종속 규칙은 분리한다: 언어/프레임워크 전용 관례(파일 블록 순서, 컴포넌트 작성 방식 등)는 이 보편 규칙과 섞지 말고, 해당 스택 스킬 또는 아래 부록으로 분리한다.
